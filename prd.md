# Info-Butler 产品需求文档（PRD）

> 版本：v1.0 | 更新日期：2026-04-20 | 作者：项目 Owner

## 一、产品概述

### 1.1 产品定位

**Info-Butler（个人情报官与行动提取器）** 是一款面向知识工作者的个人信息处理中台，核心使命是将碎片化信息转化为可执行的行动项和可检索的知识卡片，形成"输入 → 消化 → 行动 → 复盘"的完整闭环。

### 1.2 产品愿景

消灭"收藏即遗忘"的信息焦虑，让每一条有价值的信息都能落地为行动或沉淀为可检索的知识资产。

### 1.3 目标用户

| 用户画像 | 典型场景 | 核心诉求 |
|---------|---------|---------|
| 技术开发者 | 每天浏览技术博客、公众号、GitHub Issue | 快速提炼要点，提取可实践的技术行动 |
| 产品/运营 | 收藏大量行业报告、竞品分析 | 结构化归档，按主题检索历史洞察 |
| 终身学习者 | B 站课程、播客笔记、读书摘录 | 将灵感转化为待办，避免"看完就忘" |

### 1.4 核心价值主张

- **不只是总结，更要行动**：强制提取 Action Item，推动知行合一
- **不只是存储，更要检索**：标签 + 向量双重检索，按记忆线索找回知识
- **不只是工具，更要闭环**：周复盘机制量化成长，SQL 可度量行动完成率

## 二、痛点与机会分析

### 2.1 现有痛点

| # | 痛点 | 严重度 | 现有方案缺陷 |
|---|------|--------|-------------|
| P1 | 收藏即遗忘 | 高 | 书签/收藏夹无摘要，回看成本极高 |
| P2 | 看完就忘，无法转化 | 高 | 笔记工具只存不催，缺少行动追踪 |
| P3 | 难以按记忆线索检索 | 中 | 全文搜索无法匹配模糊语义回忆 |
| P4 | 信息来源分散 | 中 | 微信/浏览器/B站/播客各自为政 |
| P5 | 缺乏复盘节奏 | 中 | 无周期性提醒，知识无法内化 |

### 2.2 竞品分析

| 维度 | Notion/Obsidian | Readwise | Info-Butler |
|------|----------------|----------|-------------|
| 摘要生成 | 手动 | 自动（高亮） | AI 自动生成 |
| 行动项提取 | 无 | 无 | 强制提取 + 状态追踪 |
| 检索方式 | 全文/标签 | 全文/标签 | 标签 + 向量语义检索 |
| 复盘机制 | 手动 | Daily Review | 周报自动生成 |
| 输入便捷性 | 手动录入 | Kindle/浏览器集成 | 多渠道（API/机器人/前端） |

## 三、功能需求

### 3.1 功能架构图

```
Info-Butler
├── 信息输入层
│   ├── 文本直接输入
│   ├── URL 链接输入（自动抓取正文）
│   └── 机器人输入（微信/钉钉 Webhook）
├── AI 处理层
│   ├── 摘要生成（≤100字）
│   ├── Action Items 提取
│   ├── 标签自动生成（3-5个）
│   └── 正文向量化（Phase 2）
├── 数据存储层
│   ├── 原始信息存储
│   ├── 行动项管理
│   ├── 标签体系
│   └── 向量索引（Phase 2）
├── 检索与查询层
│   ├── 标签过滤查询
│   ├── 关键词搜索
│   └── 语义检索（Phase 2）
├── 复盘层
│   ├── 周报自动生成
│   ├── 行动项完成率统计
│   └── 知识卡片时间线
└── 前端展示层（Vue3）
    ├── 信息录入页
    ├── 知识卡片列表
    ├── 行动项看板
    └── 复盘仪表盘
```

### 3.2 功能模块详细说明

#### 模块一：信息输入（P0 - MVP 必须）

| 功能点 | 优先级 | 描述 | 验收标准 |
|--------|--------|------|---------|
| 文本输入 | P0 | 用户提交纯文本内容 | 提交后返回 task_id，状态为 processing |
| URL 输入 | P0 | 提交 URL，系统自动抓取正文 | 抓取失败标记 failed 并记录原因 |
| 输入类型标记 | P1 | 标记来源类型（text/url/voice） | 来源类型写入 raw_infos.source_type |
| 去重检测 | P2 | 相同 URL 不重复处理 | URL 相同时返回已有记录 |

#### 模块二：AI 处理（P0 - MVP 必须）

| 功能点 | 优先级 | 描述 | 验收标准 |
|--------|--------|------|---------|
| 摘要生成 | P0 | 100 字以内核心摘要 | 摘要 ≤ 100 字，覆盖核心观点 |
| Action Items 提取 | P0 | 提取可执行行动项 | 每条行动项以动词开头，可独立执行 |
| 标签生成 | P0 | 自动生成 3-5 个标签 | 标签数量 3-5，格式统一 |
| JSON 强制输出 | P0 | Dify Workflow 输出结构化 JSON | 输出符合预定义 JSON Schema |
| 异常兜底 | P0 | LLM 输出异常时进入待处理队列 | 校验失败不崩溃，记录错误原因 |

#### 模块三：行动项管理（P0 - MVP 必须）

| 功能点 | 优先级 | 描述 | 验收标准 |
|--------|--------|------|---------|
| 行动项列表 | P0 | 按状态筛选行动项 | 支持 pending/done/ignored 过滤 |
| 状态流转 | P0 | pending → done / ignored | 状态变更记录更新时间 |
| 批量操作 | P1 | 批量标记完成/忽略 | 一次操作影响多条记录 |
| 优先级标记 | P2 | 行动项标注优先级（高/中/低） | 优先级可筛选排序 |
| 截止日期 | P2 | 行动项可设置截止日期 | 过期行动项高亮提醒 |

#### 模块四：知识卡片与检索（P1）

| 功能点 | 优先级 | 描述 | 验收标准 |
|--------|--------|------|---------|
| 知识卡片列表 | P1 | 展示摘要 + 标签 + 行动项概览 | 支持分页、按时间倒序 |
| 标签过滤 | P1 | 按标签筛选知识卡片 | 多标签 AND/OR 查询 |
| 关键词搜索 | P1 | 全文关键词搜索 | 搜索摘要和原文 |
| 卡片详情 | P1 | 查看原文、摘要、行动项、标签 | 完整展示所有关联数据 |
| 语义检索 | P2 | 基于向量相似度的语义搜索 | Top-K 结果相关度 > 0.7 |

#### 模块五：复盘与统计（P1）

#### 模块六：前端界面（P1）

## 四、用户故事

### US-01：提交文章链接

> 作为知识工作者，我想通过粘贴 URL 提交一篇文章，系统自动抓取正文并生成摘要、行动项和标签，以便我无需手动阅读全文即可获取核心要点。

**验收标准：**
- 提交 URL 后 2 秒内返回 `{"task_id": "xxx", "status": "processing"}`
- 后台处理完成后，可通过 `GET /api/v1/digest/{task_id}` 获取结果
- 处理失败时，状态标记为 `failed` 并记录错误原因

### US-02：查看待办行动项

> 作为知识工作者，我想查看所有未完成的行动项列表，以便安排今天的执行计划。

**验收标准：**
- `GET /api/v1/actions?status=pending` 返回所有 pending 行动项
- 每条行动项关联原始信息摘要，可跳转查看原文
- 支持按创建时间倒序排列

### US-03：完成行动项

> 作为知识工作者，我想标记一个行动项为"已完成"，以便追踪我的执行进度。

**验收标准：**
- `PATCH /api/v1/actions/{id}` 可将状态从 pending 改为 done
- 状态变更自动记录 `updated_at` 时间
- 已完成行动项不再出现在 pending 列表

### US-04：按标签检索知识卡片

> 作为知识工作者，我想按标签（如 #FastAPI）筛选知识卡片，以便快速找到特定主题的历史积累。

**验收标准：**
- 支持单标签和多标签（AND）过滤
- 结果按时间倒序，支持分页
- 每张卡片展示摘要 + 标签 + 行动项数量

### US-05：查看周复盘报告

> 作为知识工作者，我想查看本周的知识摄入和行动完成情况，以便评估自己的学习效率和执行力度。

**验收标准：**
- 自动汇总本周新增信息条数、行动项数、完成数
- 计算行动完成率（done / (done + pending + ignored)）
- 展示本周高频标签

## 五、非功能需求

| 维度 | 指标 | 目标值 |
|------|------|--------|
| 性能 | API 响应时间（非 AI 处理） | ≤ 200ms（P95） |
| 性能 | AI 处理完成时间 | ≤ 30s（P95） |
| 可用性 | 系统可用率 | ≥ 99%（单机部署） |
| 可靠性 | AI 处理失败率 | ≤ 5%，失败有明确原因 |
| 安全性 | API 鉴权 | 所有接口需 API Key |
| 安全性 | 敏感数据 | 不记录 API Key 明文到日志 |
| 可扩展性 | AI 引擎可替换 | Dify → 任意 LLM Provider，网关层无需改动 |
| 数据 | 单条信息最大长度 | 50,000 字符 |
| 数据 | 标签数量上限 | 每条信息 3-5 个标签 |

## 六、数据流转

```
用户输入（文本/URL）
       │
       ▼
  FastAPI 网关 ──── 写入 raw_infos（status=processing）─── 返回 task_id
       │
       ▼
  Redis 任务队列（Celery/ARQ）
       │
       ▼
  异步 Worker
       │
       ├── URL? → 抓取正文（httpx + readability）
       │
       ▼
  调用 Dify Workflow API
       │
       ├── 成功 → Pydantic 校验 JSON
       │         │
       │         ├── 校验通过 → 写入 action_items + tags + 更新 raw_infos（status=done）
       │         │
       │         └── 校验失败 → 写入 raw_infos（status=parse_error），记录原始响应
       │
       └── 失败 → 写入 raw_infos（status=failed），记录错误原因
```

## 七、状态机定义

### 7.1 信息处理状态（raw_infos.status）

```
processing ──→ done
     │
     ├──→ failed        （抓取失败 / Dify 调用失败）
     │
     └──→ parse_error   （Dify 返回但 JSON 校验失败）
```

### 7.2 行动项状态（action_items.status）

```
pending ──→ done
     │
     └──→ ignored
```

## 八、里程碑与优先级

### Phase 1：MVP（核心闭环）

- [x] 文本/URL 输入接口
- [x] Dify 最简 Workflow（摘要 + 行动项 + 标签）
- [x] FastAPI 同步链路：接收 → 调 Dify → 解析 → 落库
- [x] 行动项 CRUD 接口
- [x] 基础标签查询

**验收标准：** 通过 HTTPie/Postman 可完成"输入 → AI 处理 → 查看结果 → 标记行动项完成"全流程。

### Phase 2：业务化改造

- [ ] 同步改异步（Redis 任务队列）
- [ ] 完整数据库设计（含标签多对多）
- [ ] Dify Workflow 升级（强制 JSON Schema）
- [ ] Pydantic 强校验 + 异常队列
- [ ] Vue3 前端（录入页 + 卡片列表 + 行动项看板）

**验收标准：** 可日常使用，每天往里扔内容，前端可查看和管理。

### Phase 3：高级特性

- [ ] 周复盘报告自动生成
- [ ] LangChain + 向量检索
- [ ] 微信/钉钉机器人集成
- [ ] 复盘仪表盘（统计图表）
- [ ] 深色模式

**验收标准：** 形成完整的知识管理闭环，可替代现有收藏/笔记习惯。

## 九、成功指标

| 指标 | 衡量方式 | 目标 |
|------|---------|------|
| 日活使用 | 每日提交信息条数 | ≥ 3 条/天 |
| 行动完成率 | done / total | ≥ 40% |
| 处理成功率 | done / (done + failed + parse_error) | ≥ 95% |
| 检索命中率 | 语义检索 Top-5 相关率 | ≥ 80%（Phase 3） |
| 周复盘参与率 | 每周查看周报次数 | ≥ 1 次/周 |
